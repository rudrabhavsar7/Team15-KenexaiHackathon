EXAMPLE 1 — DATABASE FAILURE (empty context)
Q: "How do I solve the issue of database failure?"
CONTEXT: (empty)
═══════════════════════════════════════════════════════
[Context empty → Context Signals omitted entirely]
[Triage-first: confirm → access → service → logs → fix]

- Intent: RESOLUTION_STEPS
- Issue Type: Database Failure / Database Server Down
- Immediate Actions (0–30 min):
  - Confirm database is actually unreachable:
    ping <db-server-ip> and attempt a test connection:
    psql -h <host> -U <user> -c "SELECT 1;" (Postgres)
    mysql -h <host> -u root -p -e "SELECT 1;" (MySQL) [SRE best practice]
  - SSH into the database server or access via
    IPMI/OOB console if SSH is unavailable [SRE best practice]
  - Check database service status:
    systemctl status postgresql
    systemctl status mysql
    systemctl status mongod [SRE best practice]
  - Read crash logs to identify failure reason:
    Postgres: /var/log/postgresql/postgresql-*.log
    MySQL: /var/log/mysql/error.log
    journalctl -u postgresql --since "1 hour ago" [SRE best practice]
  - Check system resources — OOM Killer, disk full, CPU spike
    are the 3 most common causes of sudden DB failure:
    free -h | df -h | top [SRE best practice]
- Short-Term Actions (30 min–4 hrs):
  - Restart database service once root cause is identified:
    systemctl restart postgresql [SRE best practice]
  - Check for disk space exhaustion on data partition:
    df -h /var/lib/postgresql — full disk = DB crash [SRE best practice]
  - Check for table corruption or dirty shutdown:
    Postgres: pg_resetwal (last resort) or restore from backup
    MySQL: mysqlcheck --all-databases [SRE best practice]
  - Review active connections and long-running queries
    before restart to avoid data loss:
    SELECT * FROM pg_stat_activity WHERE state = 'active'; [SRE best practice]
  - Verify replication status if using primary-replica setup:
    SELECT * FROM pg_stat_replication; [SRE best practice]
- Long-Term / Preventive Actions:
  - Set up automated DB health checks every 60 seconds
    with PagerDuty / Alertmanager alerting [SRE best practice]
  - Implement point-in-time recovery (PITR) backups and
    test restore monthly [SRE best practice]
  - Configure connection pooling (PgBouncer / ProxySQL)
    to prevent connection exhaustion crashes [SRE best practice]
  - Add disk space alerts at 70%, 85%, and 95% thresholds [SRE best practice]
- Verification & Validation:
  - Confirm DB service is running:
    systemctl status postgresql → must show active (running)
  - Run a live query: SELECT NOW(); → must return in under 1 second
  - Confirm all dependent application health checks return green
  - Verify replication lag is back to normal if replica exists

    recently dropped VPN traffic [SRE best practice]

  - Review change log: firewall updates, cert renewals,
    firmware changes in last 24 hours [SRE best practice]

  - Deploy redundant VPN gateways with automatic failover
    (active-passive or active-active) [SRE best practice]
  - Set cert expiry alerts at 30, 14, and 7 days before expiry [SRE best practice]
  - Implement synthetic VPN health checks every 5 minutes [SRE best practice]

Q: "How do I fix high CPU usage on a server?"
CONTEXT: (empty)
═══════════════════════════════════════════════════════
[Context empty → Context Signals omitted entirely]

- Intent: RESOLUTION_STEPS
- Issue Type: Server High CPU Utilization
- Immediate Actions (0–30 min):
  - Confirm CPU is actually high:
    top or htop — check %CPU column
    uptime — if load average > number of cores, system is overloaded [SRE best practice]
  - Identify the top CPU-consuming process:
    ps aux --sort=-%cpu | head -20 [SRE best practice]
  - Check if a runaway cron job, build task, or backup
    is the unexpected cause [SRE best practice]
  - Read system logs for context:
    dmesg | tail -50
    journalctl -p err --since "1 hour ago" [SRE best practice]
  - If a non-critical process is consuming > 80% CPU,
    renice or kill it:
    renice -n 10 -p <PID> or kill -9 <PID> [SRE best practice]
- Short-Term Actions (30 min–4 hrs):
  - Profile the high-CPU app:
    perf top or strace -p <PID> to find the hot code path [SRE best practice]
  - Check for CPU steal in VMs — high steal = noisy neighbor:
    vmstat 1 10 — look at the 'st' column [SRE best practice]
  - Review application logs for tight loops, infinite retries,
    or thundering herd patterns [SRE best practice]
  - Scale horizontally if traffic spike is the cause:
    spin up additional instance and load balance [SRE best practice]
- Long-Term / Preventive Actions:
  - Set CPU alerts at 80% for 5 min and 95% for 1 min [SRE best practice]
  - Implement auto-scaling triggered at CPU > 70%
    sustained for 3 minutes [SRE best practice]
- Verification & Validation:
  - Confirm top shows CPU usage returned to normal baseline
  - Verify application response time SLIs are back within SLO range

EXAMPLE 4 — DISK FULL (empty context)
Q: "How do I fix a disk full issue on a server?"
CONTEXT: (empty)
═══════════════════════════════════════════════════════
[Context empty → Context Signals omitted entirely]

- Intent: RESOLUTION_STEPS
- Issue Type: Disk Full / Storage Exhaustion
- Immediate Actions (0–30 min):
  - Confirm disk usage:
    df -h — identify which partition is at 100% [SRE best practice]
  - Find the largest files and directories immediately:
    du -sh /* 2>/dev/null | sort -rh | head -20 [SRE best practice]
  - Check for large log files that can be safely truncated:
    ls -lhS /var/log/ | head -20 [SRE best practice]
  - Truncate or rotate large logs without deleting:
    truncate -s 0 /var/log/app.log
    logrotate -f /etc/logrotate.conf [SRE best practice]
  - Delete old temp files and package caches for quick relief:
    apt clean / yum clean all
    rm -rf /tmp/* [SRE best practice]
- Short-Term Actions (30 min–4 hrs):
  - Identify deleted-but-still-open files consuming space:
    lsof | grep deleted | awk '{print $7, $9}' | sort -rn [SRE best practice]
  - Archive and compress old data to cold storage or S3 [SRE best practice]
  - Check database transaction logs or WAL files
    if DB is on this host — they grow unbounded without pruning [SRE best practice]
  - Expand the volume if cloud/VM allows:
    AWS: modify EBS → resize filesystem
    GCP: resize persistent disk [SRE best practice]
- Long-Term / Preventive Actions:
  - Set disk usage alerts at 70%, 85%, and 95% thresholds [SRE best practice]
  - Implement automated log rotation with max size caps [SRE best practice]
  - Add disk capacity trending to detect growth rate
    before it becomes critical [SRE best practice]
- Verification & Validation:
  - Confirm df -h shows partition below 80% usage
  - Verify application is writing new logs and temp files
    without errors
  - Confirm any stopped services restarted successfully
